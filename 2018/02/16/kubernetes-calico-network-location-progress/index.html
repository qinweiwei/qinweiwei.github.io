kubernetes集群中calico网络不通的问题定位过程记录

手动搭建kubernetes集群，并基于Standard Hosted Install方式搭建calico网络，具体可见上一篇博客，搭建完成后创建pod，发现网络不通，以下为定位并解决该问题的过程记录如下：

  1. 通过kubectl get pod -o wide查看pod的ip地址及所在node节点
  2. 在所在node节点查看路由信息，未发现异常
Code

  1. calico默认安装方式为ipip模式，在节点qinweiwei-1上ping pod的ip 192.168.135.3地址无法ping通，在节点qinweiwei-2上通过tcpdump命令抓包，未抓到任何包

    tcpdump -i any -nvv ip proto 4

  2. 因为环境是基于openstack的虚机搭建，于是查看虚机的secgroup规则时发现，由于secgroup验证，虚机之前ipip协议的包通过安全组，设置安全组后，发现qinweiwei-2上通过tcpdump抓包后抓到的icmp的包，但是仍然无法ping通

  3. 查看route没有问题，怀疑是iptables的原因，执行iptables -F命令后，可以ping通了，可以确定是iptables的原因导致无法ping通

  4. 依次执行iptables -nvxL INPUT ，iptables -F INPUT等类似的命令，最终确定是规则链cali-pri-k8s_ns.default造成无法ping通

Code
  1. 查看资料该自定义的链与calico的profile有关，通过calioctl命令查询profile，发现profile为空
Code
  1. 手动创建名为k8s_ns.default的proflie后，网络可以ping通
Code
  1. 可以发现造成网络不通的原因是kuberentes的ns的profile没有自动创建导致的，初步怀疑是calico-policy-controller的问题
  2. 查看calio-policy-controller的log发现controller与kuberentes api交互抱出大量的JSONDecodeError，应该是controller获取namespaces，pod出错
Code
  1. 通过命令ss命令，查看到calico-policy-controller与kuberentes的api连接已经建立
Code

从而发现连接没有问题，应该是权限的问题

  1. 最终发现是权限的问题，由于calico-policy-controller的serviceaccount设置错误，导致controller获取ns， pod等api失败，修改后问题解决。
